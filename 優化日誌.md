## 2026-04-27（第二輪 · 晚間排程）
**思考面向：** HTML 產生邏輯
**發現：** 三個空狀態 HTML（`make_grocery_section` line 229、`build_html` 餐廳 line 333、其他 line 338）以**完全相同**的字串模式重複出現，僅 icon（🛒/🍽/📦）與 label（超市/餐廳/其他）不同。每處 ~200 bytes 內聯 HTML，三處共 ~600 bytes 重複。
**改動：** 抽出 `empty_state(icon, label)` 共用函式（4 行），三個重複字串改為單行函式呼叫。`generate_static.py` 從 20077 bytes 縮短至 19906 bytes（-171 bytes），淨削減 3 處 inline HTML。
**效果：**
- 未來新增空狀態類型（譬如「健康」「娛樂」分頁）時，從「複製 ~200 bytes 內聯 HTML 並小心改 emoji/label」縮短為「呼叫 `empty_state("🏥", "健康")` 一行」。
- 改空狀態樣式（hint 文案、顏色、字級）時，從「3 個地方各改一次（容易漏改）」縮短為「改 1 處」。
- HTML 輸出 byte-for-byte 完全相同（已用 `diff` 驗證，唯一差異是時間戳）。

## 2026-04-27
**思考面向：** 資料結構
**發現：** `GROCERY` 與 `DATA` 中每筆記錄都帶有 `"id":N` 欄位（共 17 筆 = 4 grocery + 11 sienna + 1 c300），但 grep 全檔案後確認 **`id` 在程式碼中從未被讀取**（沒有 `r["id"]`、`.get("id")`、或任何引用）。更糟的是，這些 ID **不唯一**：grocery 用 1-4、sienna 用 1-11（其中 1-4 與 grocery 重複）、c300 用 1（與 sienna、grocery 都重複）。純粹的雜訊欄位，每筆新收據都得多寫一個沒有用、還會搞混的編號。
**改動：** 用 `sed -E 's/"id":[0-9]+, +//g'` 一次移除所有 17 筆記錄的 `id` 欄位。`generate_static.py` 從 20217 bytes 縮短至 20077 bytes（-140 bytes）。
**效果：**
- 未來 receipt-processor 新增收據時，從「決定一個 id 編號（還得查目前最大值避免衝突）+ 寫進資料」縮短為「直接寫進資料」，每筆記錄少一欄。
- 移除了「ID 重複但程式照樣運作」這個誤導性陷阱——以後若真要加唯一識別碼，就乾乾淨淨地從零設計，不會被舊的假 ID 干擾。
- HTML 輸出 byte-for-byte 完全相同（已用 `diff` 驗證，唯一差異是時間戳）。

## 2026-04-25
**思考面向：** 圖表函式
**發現：** `_line_svg` 簽章有 `color_dot` 參數，但檢查 5 個呼叫點（`make_price_svg` x2、`make_eff_svg` x1、`make_spending_svg` x2，含空白 placeholder），`color_dot` **永遠等於 `color_line`**。100% 冗餘參數，每個呼叫點都重複指定同一個十六進位色碼字串。
**改動：** 從 `_line_svg` 簽章移除 `color_dot`；資料點 `<circle>` 直接用 `color_line` 上色。5 個呼叫點各刪除一個冗餘色碼字串。`generate_static.py` 從 20282 bytes 縮短至 20217 bytes（-65 bytes）。
**效果：**
- 未來新增圖表時，從「指定 4 個顏色（line/fill/dot/label）」縮短為「指定 3 個顏色（line/fill/label）」，少一個容易寫錯的同色字串。
- 改主色時，從「同步改 line + dot 兩處（容易漏改）」縮短為「改 1 處」。
- HTML 輸出 byte-for-byte 完全相同（已用 `diff` 驗證，唯一差異是時間戳）。
