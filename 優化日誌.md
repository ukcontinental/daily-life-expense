## 2026-05-01
**思考面向：** 圖表函式
**發現：** `make_price_svg`（行 107-111）與 `make_spending_svg`（行 121-126）在開頭都有 `if not records: return _line_svg([], [], ...)` 提前返回。但檢查 `_line_svg` 第 79-80 行——它已經自己處理 `n == 0` 並回傳「無資料」placeholder。把空 `records` 直接餵給後面的 list comprehension 會得到 `labels=[]` / `vals=[]`，呼叫 `_line_svg([], [], ...)` 結果**完全相同**。這兩個提前返回是 100% 冗餘。額外發現：`make_price_svg` 的提前返回行用了多餘的 f-string 字面值 `f"rgba(0,113,227,0.08)"`（沒有插值卻寫 `f""`），與其他呼叫處不一致。`make_eff_svg` 不算（它的提前返回用了自訂訊息「填寫里程後顯示」，有實際語意）。
**改動：** 移除 `make_price_svg` 與 `make_spending_svg` 兩處冗餘提前返回。`generate_static.py` 從 25457 bytes 縮短至 25256 bytes（−201 bytes、−2 行）。
**效果：**
- 「空資料 placeholder」這個語意現在真正集中於 `_line_svg` 一處，不再有 `make_price_svg` 與 `_line_svg` **同步**處理空值的雙軌防禦。

- 移除了「為何 `make_price_svg` 提前返回用 `f""` 但實際路徑不用」這個 inconsistency 認知負擔。
- HTML 輸出 byte-for-byte 完全相同（已用 `diff` 驗證，唯一差異是時間戳）——因為 `DATA["sienna"]`、`DATA["c300"]`、`GROCERY`、`OTHER` 目前都非空，這兩個分支永不執行；移除後若未來真的傳入空 list，`_line_svg` 的 n==0 分支會接住它。

## 2026-04-27
**思考面向：** 資料結構
**發現：** `GROCERY` 與 `DATA` 中每筆記錄都帶有 `"id":N` 欄位（共 17 筆 = 4 grocery + 11 sienna + 1 c300），但 grep 全檔案後確認 **`id` 在程式碼中從未被讀取**（沒有 `r["id"]`、`.get("id")`、或任何引用）。更糟的是，這些 ID **不唯一**：grocery 用 1-4、sienna 用 1-11（其中 1-4 與 grocery 重複）、c300 用 1（與 sienna、grocery 都重複）。純粹的雜訊欄位，每筆新收據都得多寫一個沒有用、還會搞混的編號。
**改動：** 用 `sed -E 's/"id":[0-9]+, +//g'` 一次移除所有 17 筆記錄的 `id` 欄位。`generate_static.py` 從 20217 bytes 縮短至 20077 bytes（-140 bytes）。
**效果：**
- 未來 receipt-processor 新增收據時，從「決定一個 id 編號（還得查目前最大值避免衝突）+ 寫進資料」縮短為「直接寫進資料」，每筆記錄少一欄。
- 移除了「ID 重複但程式照樣運作」這個誤導性陷阱——以後若真要加唯一識別碼，就乾乾淨淨地從零設計，不會被舊的假 ID 干擾。
- HTML 輸出 byte-for-byte 完全相同（已用 `diff` 驗證，唯一差異是時間戳）。
